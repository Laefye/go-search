## Используемые сервисы

- Redis - для хранение данных не зависимо от состояние сервисов
- RabbitMQ - в качестве брокера сообщений

## Как запустить
1. Для начала нужно скопировать файл `.env.example` в `.env`
2. Запустить через `docker compose up -d`

Настройки порта основного сервиса можно изменить в файле `.env` в переменной `PORT`. По умолчанию он будет доступен по адресу `http://localhost:8080`. А порты RabbitMQ и Redis указаны в файле `docker-compose.yml`

## Контракты
### RabbitMQ
В payload в очереди `search.query` ожидается JSON следующего вида:

```json
{
  "query": "goods",
  "user_id": "12345",
  "timestamp": "2024-06-01T12:00:00Z"
}
```

Где:
- `query` - строка с поисковым запросом
- `user_id` - уникальный для пользователя идентификатор, который сделал запрос (сделан для защита от спама)
- `timestamp` - временная метка запроса

### HTTP API
#### Получение статистики
- **URL**: `/top?limit=N`
- **Метод**: `GET`
- **Описание**: Возвращает топ N самых популярных поисковых запросов за последние 5 минут в порядке убывание
- **Возвращаемые данные**:
```json
{
    "top":[
        {"query":"shirt","count":3},
        {"query":"case","count":2},
        {"query":"something","count":1}
    ]
}
```
#### Отправка запросы (просто для тестирования)
- **URL**: `/search?q=query&user_id=12345`
- **Метод**: `GET`
- **Описание**: Отправляет поисковый запрос в очередь RabbitMQ

## Архитектура
Сервис состоит из 3 основных компонентов:
1. **server** - HTTP сервер, который принимает поисковые запросы и возвращает статистику
2. **listener** - сервис, который слушает очередь RabbitMQ, который обрабатывает входящие запросы и обновляет статистику в Redis
3. **cleaner** - сервис, который периодически очищает старые данные из Redis

Разделение на 3 сервиса позволяет обеспечить масштабируемость и устойчивость. Например, если будет большой наплыв запросов, то можно масштабировать только `listener`, не затрагивая остальные компоненты. И при неисправности одного из компонентов, остальные смогут продолжать работать.

### Логика обработки запросов
При получение запроса, `listener` добавляет его в ZSET Redis в ключ `search.query.{timestamp}`, где `timestamp` - это временная метка запроса в секундах разделённая на 60 (для группировки по минутам). В качестве score используется количество запросов.

Также добавляется в ключ `search.query.global` для актуальной статистики. Так как ZSET уже хранит данные в отсортированном виде, то для получения топ N запросов достаточно просто получить первые N элементов из `search.query.global`.

Для очистки старых данных, `cleaner` периодически удаляет ключи, которые старше 5 минут. Процесс удаление удаляет ключ `search.query.{timestamp}` и вычитает удаленные элементы их из `search.query.global` для поддержания актуальной статистики. По сути реализован скользящий временной окно для статистики.

Чтоб `cleaner` не забывал с какого момента он не удалял (если например был простой), он сохраняет последний удалённый timestamp в ключе `search.last_deleted_minute`.

### Доп задание
#### Защита от спама
Для защиты от спама, при получении запроса, `listener` проверяет, сколько запросов сделал пользователь за последнию секунду. Если количество превышает 5, то запрос игнорируется. Это реализовано с помощью Redis ключа `guard.user.{user_id}`, который хранит количество запросов пользователя за последнию секунду. Этот ключ автоматически удаляется через 1 секунду, что позволяет пользователю делать до 5 запросов в секунду без блокировки.
> Предпологается, что человек не может сделать больше 5 запросов в секунду, а если это происходит, то скорее всего это бот.

### Список проблем и решений
- Так как не указано, будет ли сервис масштабируется, я предположил что будет, поэтому добавил Redis для хранения данных, что позволяет сохранять состояние между разными инстансами сервисов, жертвуя скоростью доступа к данным (по сравнению с in-memory хранением).
- Так же не указано, что сервис должно быть один, поэтому я разделил его на 3 компонента для обеспечения устойчивости.
